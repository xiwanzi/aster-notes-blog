# 架构分析与实现说明

## 1. 参考站的界面骨架

参考站不是传统“标题 + 文章列表”的博客，而是三层叠放的数字花园首页：

1. 固定在顶部的胶囊导航负责全局入口与显示偏好。
2. 接近一屏高的沉浸式 Hero 负责品牌表达，背景图、网格和渐变文字构成第一视觉焦点。
3. Hero 上方叠放桌面三栏信息区：左栏是作者、公告、音乐和分类，中栏是文章流，右栏是统计和日历。移动端隐藏两侧信息密度，只保留 Hero、分类滑条和文章卡片。

高还原的关键不是复制单个像素，而是保持这些比例关系：顶部导航宽度约等于内容网格；桌面中栏约为侧栏的 2.7 倍；Hero 内容居中但两侧信息卡同时可见；文章区延续同一组圆角、描边、阴影与强调色。

## 2. 当前实现架构

项目使用 Next.js App Router，但首页仍保持静态数据驱动。这样既能被 Sites 的构建与部署流程直接接收，也为以后增加独立文章页、Markdown/MDX、RSS 和服务端元数据留下自然扩展点。

```text
内容数据 site.ts
      ↓
BlogHome 客户端组合组件
      ↓
全局设计令牌 + 响应式布局
      ↓
Next.js 页面与 Sites 部署产物
```

`site.ts` 是唯一的内容源。界面组件只解释数据，不保存文案副本。交互状态分为两类：主题与磨砂偏好写入 `localStorage`；搜索、菜单和音乐播放只保留在当前会话。这样避免在内容尚未确定时引入数据库或 CMS 的维护成本。

## 3. 视觉系统

- 主色：粉色 `#ee5da8`，辅色：青色 `#1cc6cc`。
- 背景与卡片使用暖白而非纯白，暗色模式使用深灰紫而非纯黑。
- 组件共享 11–15px 圆角、半透明描边和两级阴影。
- Hero 使用原创 SVG、CSS 渐变和网格纹理，不依赖第三方图片请求。
- 字体使用系统字体栈，不阻塞首屏渲染，也避免中文字体跨区加载的不稳定性。

## 4. 响应式策略

- `> 1040px`：完整三栏，侧栏叠在 Hero 上。
- `761–1040px`：隐藏侧栏，中栏保持较宽的阅读尺寸，并与 Hero 轻微叠放。
- `<= 760px`：单栏顺序流，Hero 高度压缩，分类条可横向滚动，导航改为图标与菜单。
- `<= 480px`：进一步压缩按钮和文章卡片密度，搜索结果改为两列。

主要触摸目标保持约 40–46px，文章正文行高和对比度针对手机阅读单独调整。

## 5. 已完成优化

- 零运行时 UI 依赖，图标为内联 SVG，减少 JS 与网络请求。
- Hero 为本地矢量资源，清晰度与文件体积不随屏幕分辨率急剧增长。
- 动画只使用 `transform`、`opacity` 和宽度进度反馈；系统要求低动态时全部降级。
- 主题和磨砂效果均可持久化，暗色模式通过设计令牌统一切换。
- 搜索在浏览器内完成，不上传查询内容；Escape 与点击遮罩均可关闭。
- 语义化使用 `nav`、`main`、`section`、`article`、`aside`、`footer`，关键控件含可访问名称与键盘焦点样式。
- 没有复制参考站的作者信息、文章正文或受版权保护的背景素材。

## 6. 推荐的下一阶段

内容方向确定后，优先增加 `posts/[slug]` 文章页和 MDX 内容目录；文章超过约 30 篇时再引入静态内容索引。评论、登录、数据库和完整 CMS 都不应在内容模型稳定前加入，否则会增加不必要的部署与维护复杂度。
